## 一、系统架构

### 1.1 三层架构

```
┌─────────────────────────────────────────────────────────┐
│                    应用层（第三层）                      │
│   电子沙盘 │ 告警闭环管理 │ 专家系统（3个独立前端）          │
├─────────────────────────────────────────────────────────┤
│                    AI平台层（第二层）                      │
│   ai-vision │ ai-video │ ai-voice │ ai-coordinator    │
├─────────────────────────────────────────────────────────┤
│                    感知层（第一层）                       │
│   sensor-collector │ video-streamer │ drone-integration │
│                      │ gateway（统一接入）               │
└─────────────────────────────────────────────────────────┘
```

### 1.2 技术栈

| 层级 | 技术 |
|------|------|
| 后端API | Python / FastAPI / SQLAlchemy / PostgreSQL |
| 时序数据 | TimescaleDB（PostgreSQL扩展）|
| 消息队列 | Redis Stream |
| AI推理 | YOLOv8 / ONNX Runtime / Whisper / ByteTrack |
| 前端 | React / TypeScript / CesiumJS / Ant Design |
| 部署 | Docker / Docker Compose |
| 向量库 | Milvus（RAG知识库）|

### 1.3 关键接口

| 接口 | 协议 | 说明 |
|------|------|------|
| 感知层→AI平台 | REST/Redis Stream | 传感器MQTT/视频流RTSP |
| AI平台→应用层 | REST/WebSocket | 检测结果/告警推送 |
| 应用层内部 | REST | 前后端分离 |

## 二、多Agent协作机制

### 2.1 角色定义

| 角色 | 职责 | 产出物 |
|------|------|--------|
| **架构师（Architect）** | 设计拓扑、接口协议、技术选型 | 架构文档、接口协议 |
| **代码工程师（Code Engineer）** | 按架构实现具体模块 | 可运行代码 |
| **代码审查工程师（Code Reviewer）** | 独立审查代码质量 | 审查报告 |
| **测试工程师（QA Engineer）** | 编写测试、联调验证 | 测试报告 |
| **项目管家（Orchestrator/我）** | 派活、追踪、协调、验收 | 进度状态、版本规划 |

### 2.2 工作流程

```
用户需求
    ↓
项目管家：拆解任务、分配模块
    ↓
各Agent按序/并行执行（每个Agent必须）：
    1. 在 coordination.md 登记模块持有
    2. 完成工作
    3. 提交到 GitHub
    4. 删除持有记录
    ↓
项目管家：验收、合并到 main、分发下一轮任务
```

### 2.3 协调文件体系

```
ORCHESTRATOR/
├── coordination.md    # 模块持有登记（谁在改什么）
├── progress.md        # 进度追踪（任务状态一览）
└── version.md         # 版本规划（v0.1~v1.0路线图）
```

### 2.4 版本演进规则

| 版本 | 目标 | 验收标准 |
|------|------|---------|
| v0.1 | 基础框架搭建 | 所有模块有代码框架，可启动 |
| v0.2 | 联调测试 | docker-compose up 全链路可跑通 |
| v0.3 | AI能力增强 | YOLOv8/Whisper 真实推理 |
| v0.4 | 专家系统 | RAG + 表格生成可用 |
| v1.0 | 生产就绪 | 安全加固 + 性能达标 + 文档齐全 |

### 2.5 问题优先级

| 级别 | 定义 | 处理时间 |
|------|------|---------|
| P0 | 安全漏洞或致命Bug | 立即修复 |
| P1 | 功能缺陷或性能问题 | 24小时内 |
| P2 | 代码优化或体验改进 | 计划迭代 |

## 三、Git分支策略

```
main（保护分支）
  ├── 所有人可读，只有管家可合并
  ├── 每次合并需要 CI 通过
  │
  └── feature/*（功能分支）
        └── Agent 工作时创建
        └── 完成后提 PR
        └── 管家 review 后合并
```

## 五、通讯规则

1. **每次派活**：管家登记 coordination.md → 派 Agent → Agent 完成后汇报
2. **冲突处理**：先查 coordination.md，同模块不能并发
3. **验收标准**：管家验收通过才能合并 main
4. **问题升级**：Agent 遇到无法解决的阻塞，汇报管家协调
